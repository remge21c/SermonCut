# SermonCut 최종 기획서

## 1. 최종 방향

SermonCut은 하나의 Windows 로컬 실행 앱 안에 두 가지 작업 모드를 제공하는 설교 영상 반자동 편집 앱이다.

```text
SermonCut
  ├─ A. Quick Shorts
  │   유튜브 링크 또는 로컬 영상 → 자막 확보 → AI 쇼츠 후보 → 3개 선택 → 세로 쇼츠 MP4
  │
  └─ B. 본문영상 편집
      원본 영상 → 멀티 컷 → 썸네일 인트로 → 본문 편집본 MP4
      → 본문 편집본 기준 쇼츠 후보 → 3개 선택 → 세로 쇼츠 MP4
```

앱을 A와 B로 따로 분리하지 않는다.  
대신 하나의 공통 엔진을 만들고, 그 위에 두 개의 워크플로 화면을 둔다.

---

## 2. 제품 정의

SermonCut은 설교 영상을 빠르게 쇼츠로 만들거나, 본문 영상을 먼저 편집한 뒤 쇼츠까지 생성할 수 있는 Windows 로컬 실행 반자동 영상편집 앱이다.

이 앱은 CapCut 같은 풀 타임라인 편집기를 목표로 하지 않는다.  
매주 반복되는 설교 영상 편집 작업을 줄이기 위해, 자막 분석·AI 구간 추천·쇼츠 렌더링을 자동화하고, 사용자는 필요한 지점만 검수한다.

---

## 3. 핵심 원칙

## 3.1 앱과 AI의 역할 분담

```text
앱이 담당하는 일
- 영상 입력
- 자막 확보
- 구간 자르기
- 영상 이어붙이기
- 썸네일 인트로 붙이기
- 세로 9:16 렌더링
- 자막 번인
- 최종 MP4 출력

AI가 담당하는 일
- 설교 구간 분석
- 축도 구간 분석
- 설교 요약
- 키워드 추출
- 쇼츠 후보 제안
- 훅 문구 생성
- 제목/해시태그/강조문장 생성
```

AI는 영상을 직접 편집하지 않는다.  
AI는 JSON 형태의 분석 결과와 제안만 만든다.  
실제 편집과 렌더링은 로컬 앱과 FFmpeg 엔진이 수행한다.

## 3.2 공통 엔진 재사용

A Quick Shorts와 B 본문영상 편집은 별도 앱이 아니다.  
두 모드는 다음 공통 엔진을 공유한다.

```text
sermoncut-core
  ├─ 입력 처리
  ├─ 자막 확보
  ├─ Whisper 전사
  ├─ VTT/SRT 파싱
  ├─ Gemini 분석
  ├─ 쇼츠 후보 생성
  ├─ edit-spec JSON 생성
  ├─ FFmpeg 렌더링
  └─ 자막 번인
```

## 3.3 MVP 범위 (1차 = Quick Shorts)

1차 출시는 **Quick Shorts 모드만** 포함한다. 본문영상 편집(B 모드)은 2차로 분리한다.

| 구분 | 항목 |
|---|---|
| **Must (1차 필수)** | 유튜브 링크 입력 · 로컬 MP4 입력 · 자막 확보(YouTube 자막 우선 → Whisper 폴백) · 설교 구간 분석(시작/끝/축도) · 쇼츠 후보 5개 제안 · 후보 중 3개 선택 · 세로 9:16 렌더(고정 기본 템플릿 1종) · 기본 자막 번인(흰 글씨+검은 외곽선) · 메타데이터 출력(제목/훅/강조문/해시태그) |
| **Should (여유 시)** | 크롭 위치 선택(좌/중/우) · 기존 SRT/VTT 불러오기 · 분석 결과 구간 미세 확인 |
| **Could (3차 이후)** | 자막 스타일 템플릿 다중화 · 키워드 금색 강조 · 훅 자막 강화 · 자동 크롭 추천 |
| **Won't (이번 버전 OUT)** | 본문영상 멀티컷·썸네일 인트로(2차) · 자동 얼굴/화자 추적 · 자동 업로드 · 글풍선/도형 자유 배치 · BGM/볼륨 정규화/무음 제거 |

## 4. 작업 모드 A: Quick Shorts

## 4.1 목적

Quick Shorts는 가장 빠르게 쇼츠만 만드는 모드다.

사용자는 유튜브 링크 또는 로컬 영상을 넣고, 앱은 자동으로 자막을 확보한 뒤 AI가 쇼츠 후보 5개를 제안한다. 사용자는 그중 3개를 선택하고, 앱은 세로 쇼츠 MP4 3개를 만든다.

## 4.2 Quick Shorts 흐름

```text
유튜브 링크 또는 로컬 영상 입력
  ↓
영상/자막 확보 (YouTube 자막 우선 → 없으면 Whisper 폴백)
  ↓
설교 시작·끝 분석 (설교규칙.md)
  ↓
설교 본문 transcript 생성
  ↓
AI 쇼츠 후보 5개 제안 (쇼츠규칙.md)
  ↓
사용자가 3개 선택
  ↓
세로 9:16 크롭 설정 (고정 기본 템플릿 1종)
  ↓
자막 번인
  ↓
쇼츠 MP4 3개 출력
```

## 4.3 Quick Shorts 1차 포함 기능

| 구분 | 기능 | 설명 |
|---|---|---|
| Must | 유튜브 링크 입력 | 유튜브 영상에서 자막 또는 영상을 확보한다. |
| Must | 로컬 영상 입력 | MP4 파일을 직접 선택할 수 있다. |
| Must | 자막 확보 | YouTube 자막(VTT) 우선, 없으면 Whisper 전사로 폴백한다. |
| Must | 설교 구간 분석 | 설교 시작·끝, 축도 시작·끝을 찾는다. |
| Must | 쇼츠 후보 5개 | AI가 후보 5개를 제안한다. |
| Must | 쇼츠 3개 선택 | 사용자가 후보 중 3개를 선택한다. |
| Must | 세로 쇼츠 렌더 | 선택한 3개를 9:16 MP4로 출력한다. |
| Must | 기본 자막 번인 | 흰 글씨 + 검은 외곽선 자막을 영상에 넣는다. |
| Must | 메타데이터 출력 | 제목, 훅 문구, 강조문장, 해시태그를 저장한다. |

## 4.4 Quick Shorts 1차 제외 기능

| 제외 기능 | 이유 |
|---|---|
| 본문영상 멀티 컷 | B 모드에서 처리한다. |
| 썸네일 인트로 | B 모드에서 처리한다. |
| 글풍선/도형 자유 배치 | 1차 범위가 커진다. |
| 자동 얼굴 추적 | 정확도와 속도 문제가 있다. |
| 자동 업로드 | 인증과 실수 업로드 위험이 있다. |

## 5. 작업 모드 B: 본문영상 편집

## 5.1 목적

본문영상 편집 모드는 원본 설교 영상에서 필요한 여러 구간을 선택해 하나의 본문 편집본을 만들고, 그 본문 편집본을 기준으로 쇼츠를 생성하는 모드다.

이 모드는 1차 이후 2차 목표로 개발한다.

## 5.2 본문영상 편집 흐름

```text
원본 영상 입력
  ↓
자막 확보
  ↓
설교 시작·끝 분석
  ↓
본문에 넣을 여러 구간 선택
  ↓
썸네일 이미지 인트로 추가
  ↓
본문 편집본 MP4 생성
  ↓
원본↔본문 편집본 타임코드 매핑
  ↓
본문 편집본 기준 쇼츠 후보 5개 제안
  ↓
사용자가 3개 선택
  ↓
세로 쇼츠 MP4 3개 출력
```

## 5.3 B 모드 포함 기능

| 구분 | 기능 | 설명 |
|---|---|---|
| 2차 Must | 멀티 구간 선택 | 원본에서 여러 구간을 선택한다. |
| 2차 Must | 구간 이어붙이기 | 선택 구간을 순서대로 이어붙인다. |
| 2차 Must | 썸네일 인트로 | 이미지 파일을 영상 앞에 2~5초 붙인다. |
| 2차 Must | 본문 편집본 생성 | 본문영상 MP4를 출력한다. |
| 2차 Must | time_map 생성 | 원본 시간과 본문 편집본 시간을 매핑한다. |
| 2차 Must | 본문 기준 쇼츠 제안 | 본문 편집본의 transcript를 기준으로 쇼츠 후보를 만든다. |

## 5.4 B 모드에서 중요한 점

본문영상 편집을 하면 원본 시간과 본문 편집본 시간이 달라진다.  
따라서 `time_map`이 반드시 필요하다.

예:

```text
원본 00:20:00 ~ 00:35:00
원본 00:42:00 ~ 00:55:00

↓

본문 편집본 00:00:00 ~ 00:15:00
본문 편집본 00:15:00 ~ 00:28:00
```

쇼츠 후보가 본문 편집본 기준 `00:16:10 ~ 00:16:35`라면, 실제 원본 기준으로는 `00:43:10 ~ 00:43:35`일 수 있다.

이 매핑이 없으면 자막 싱크, 미리보기, 재렌더링, 쇼츠 구간 표시가 모두 꼬일 수 있다.

## 7. 데이터 모델

## 7.1 Normalized Source

```json
{
  "source": {
    "id": "src_001",
    "type": "youtube",
    "url": "https://youtube.com/watch?v=...",
    "video_path": "cache/source.mp4",
    "title": "2026년 주일예배 설교",
    "duration_sec": 5400
  },
  "transcript": [
    {
      "start": 1536.42,
      "end": 1539.81,
      "text": "오늘 본문은 요한복음 6장입니다"
    }
  ]
}
```

## 7.2 Sermon Analysis Result

`설교규칙.md`의 출력 JSON과 키가 1:1로 일치한다.
AI는 시각을 자막 타임스탬프 그대로 **"HH:MM:SS"** 문자열로 내보내고,
엔진이 `to_seconds`로 **초 단위 float**로 변환해 아래 `sermon_analysis.json`에 저장한다.

```json
{
  "part": "2부",
  "sermon_start": 1536.0,
  "sermon_end": 4820.0,
  "start_reason": "본문 멘트 시작",
  "end_reason": "마무리 기도 후 찬송 이어짐",
  "benediction_start": 4980.0,
  "benediction_end": 5105.0,
  "summary": "설교 핵심 요약",
  "keywords": ["은혜", "믿음", "기도", "말씀"]
}
```

- `benediction_start`, `benediction_end`는 축도 구간을 못 찾으면 `null`.

## 7.5 Shorts Candidate Spec

`쇼츠규칙.md`의 출력 JSON과 키가 1:1로 일치한다.
입력은 설교 본문(`sermon_start` ~ `sermon_end`)만 대상으로 하며 **축도 구간은 제외**한다.
AI는 **후보 5개**(권장 분배: 후킹형 2 · 예화형 2 · 핵심선언형 1)를 만들고,
그중 점수가 높고 성격이 다른 **3개**를 `selected_candidate_ids`로 추천한다(최종 선택은 사용자).
각 후보 길이는 기본 15~30초, 문장 완결을 위해 부득이하면 최대 40초까지 허용한다.
시각은 "HH:MM:SS"로 출력되고 엔진이 초 단위 float로 변환해 저장한다.

```json
{
  "summary": "본 설교의 핵심 메시지 요약",
  "keywords": ["은혜", "믿음", "기도"],
  "candidates": [
    {
      "id": "cand_001",
      "type": "후킹형",
      "start": 1880.0,
      "end": 1907.0,
      "duration_sec": 27.0,
      "title": "끝이라고 생각한 그 순간",
      "hook_line": "정말 끝이라고 생각했습니까?",
      "highlight": "끝이 아니라 시작입니다",
      "hashtags": ["#설교", "#믿음", "#은혜"],
      "reason": "도입 질문이 강하고 독립된 쇼츠로 완결성이 있음",
      "score": 88
    }
  ],
  "selected_candidate_ids": ["cand_001", "cand_003", "cand_005"]
}
```

B 모드에서는 후보의 `start`, `end`가 본문 편집본 기준 시간이 될 수 있다.  
그 경우 `source_start`, `source_end`를 함께 저장한다.

```json
{
  "id": "cand_001",
  "base_start": 120.5,
  "base_end": 148.2,
  "source_start": 1653.5,
  "source_end": 1681.2
}
```

## 8. 화면 구성

## 8.1 첫 화면

```text
SermonCut

무엇을 만들까요?

[ A. Quick Shorts ]
유튜브 링크나 영상 파일로 쇼츠만 빠르게 만들기

[ B. 본문영상 편집 ]
원본 영상을 여러 구간으로 편집한 뒤 쇼츠까지 만들기
```

## 8.2 A Quick Shorts 화면

```text
1. 영상 입력
   - YouTube URL
   - 로컬 MP4

2. 자막 확보 방식
   - YouTube 자막 사용
   - Whisper 전사 사용
   - 기존 SRT/VTT 불러오기

3. 분석 실행

4. 쇼츠 후보 5개 표시
   - 제목
   - 구간
   - 훅 문구
   - 강조문장
   - 해시태그
   - 이유
   - 점수

5. 3개 선택

6. 렌더 설정
   - 세로 9:16
   - 크롭 위치: 좌/중/우
   - 자막 스타일: 기본 템플릿

7. 렌더 시작

8. 결과 확인
   - short_001.mp4
   - short_002.mp4
   - short_003.mp4
```

## 8.3 B 본문영상 편집 화면

```text
1. 원본 영상 입력

2. 자막 확보

3. 설교 구간 확인

4. 본문에 넣을 구간 선택
   - 시작/끝 지정
   - 구간 추가
   - 구간 삭제
   - 순서 변경

5. 썸네일 인트로 설정
   - 이미지 선택
   - 지속 시간 설정

6. 본문 편집본 렌더

7. 본문 편집본 기준 쇼츠 후보 5개 생성

8. 3개 선택

9. 쇼츠 렌더
```

## 8.4 세로 렌더·자막 번인 사양 (1차 고정 템플릿)

1차는 **고정 기본 템플릿 1종**만 지원한다. (기존 `쇼츠_기획서.md`의 사양을 본 문서에 흡수)

```text
출력 영상
- 해상도: 1080 x 1920 (9:16)
- fps: 30
- 포맷: mp4 (H.264 / AAC)
- 크롭: 원본 가로 영상의 중앙을 9:16으로 크롭 (Should: 좌/중/우 선택)

자막 번인
- 본문 자막: 흰 글씨 + 검은 외곽선
- 위치: 화면 하단 안전영역 (하단에서 약 12~18% 여백)
- 폰트: 굵은 고딕 계열, 가독성 우선 크기 (1080폭 기준 본문 약 48~60px)
- hook_line: 영상 맨 앞 3초 상단/중앙에 강조 표시
- highlight: 핵심 한 문장을 크게 강조 표시

1차 제외 (3차 이후 = Could)
- 키워드 금색 강조, 자막 스타일 다중 템플릿, 글풍선/도형
```

## 9. 기술 스택

## 9.1 권장 스택

```text
Electron + React + Python Engine + FFmpeg
```

| 구성 | 역할 |
|---|---|
| Electron | Windows 데스크톱 앱 패키징 |
| React | UI 화면 구성 |
| Python Engine | 자막 처리, AI 분석, spec 생성, 렌더 명령 생성 |
| FFmpeg | 자르기, 붙이기, 자막 번인, 세로 렌더링 |
| Whisper | 로컬 음성 전사 |
| Gemini API | 설교 분석, 쇼츠 후보 추천 (1차 확정 엔진) |

> AI 분석 엔진은 1차에서 **Gemini API로 확정**한다. 호출부는 `call_gemini` 한 곳에 모아
> 추후 다른 LLM으로 교체할 여지만 남긴다(별도 추상화는 1차 범위 밖).

## 9.2 왜 이 구조인가

- 기존 Python 분석 로직을 재사용하기 좋다.
- FFmpeg와 Whisper를 Python에서 제어하기 쉽다.
- React/Electron은 화면 구성과 파일 선택, 프리뷰 UI에 유리하다.
- 향후 Konva/Fabric.js 기반 오버레이 편집으로 확장하기 쉽다.

## 13. 핵심 리스크와 대응

| 리스크 | 대응 |
|---|---|

| 앱이 너무 복잡해질 수 있음 | A Quick Shorts를 먼저 완성하고 B는 2차로 분리 |
| 자막/오버레이 합성이 복잡할 수 있음 | 1차는 기본 자막 번인만 지원 |

## 14. 최종 결론

SermonCut은 하나의 Windows 로컬 앱으로 개발한다.  
앱 안에는 두 가지 모드를 둔다.

```text
A. Quick Shorts
- 빠르게 쇼츠만 만드는 모드
- 1차 개발 목표

B. 본문영상 편집
- 원본 영상을 멀티 컷해 본문 편집본을 만들고 쇼츠까지 만드는 모드
- 2차 개발 목표
```

두 모드는 별도 앱으로 나누지 않고, 하나의 `sermoncut-core` 엔진을 공유한다.  
먼저 A Quick Shorts를 완성해 실제 사용 가능한 결과물을 만들고, 그 다음 B 본문영상 편집을 추가한다.
